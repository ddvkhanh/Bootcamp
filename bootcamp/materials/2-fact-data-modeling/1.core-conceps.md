## What is a Fact?

**Definition**: A fact is something that **happened or occurred** (an event)

**Examples**:

- User logs in to an app
- Transaction is made
- Run a mile with Fitbit

**Key characteristic**: Facts are **not slowly changing** → easier to model than dimensions in some respects

---

## ⚠️ What Makes Fact Modeling Hard?

### Volume Challenge

**Fact data is 10-100x the volume of dimension data**

**Example**: Facebook had 2B active users but sent **50B notifications every day**

- Dimension: 2B users
- Fact: 50B events (25x more)

### Context Challenge

Fact data needs **a lot of context** for effective analysis

### Data Quality Challenge

**Duplicates in facts are way more common** than in dimensional data

---

## 🏗️ How Fact Modeling Works - The 5 W's Framework

| Field Type | How to Model | Example |
| --- | --- | --- |
| 🧑 **Who** | Push out as IDs (not entire objects) | `user_id`, not full user object |
| 📍 **Where** | Model as IDs to join, OR denormalize if high cardinality | `device_id`, `location_id` |
| 🔧 **How** | Similar to "Where" — often denormalized | "He used an iPhone to make this click" |
| 📝 **What** | Fundamentally part of the fact nature | `GENERATED`, `SENT`, `CLICKED`, `DELIVERED` |
| ⏰ **When** | Fundamentally part of the fact nature | `event_timestamp`, `event_date` |

## 🔄 When to Denormalize?

### Real Example: Netflix Network Logs Pipeline

**Goal**: See which microservice app each network request came from/went to

**Traditional approach (normalized)**:

- Store IP address as identifier
- JOIN to populate "app" field
- **Problem**: JOIN WAS EXPENSIVE

**Solution (denormalized)**:

- Log the "app" field directly with each network request
- Get rid of the join entirely
- **✅ DENORMALIZATION SAVES THE DAY**

**Principle**: When joins become too expensive on high-volume fact data, denormalize by bringing dimension attributes into the fact table.

## 📝 Logging Best Practices

### Why Logging Matters

Logging brings in **all critical context** for fact data

### Best Practices

- Done in collaboration with **online system engineers**
- **Don't log everything!** → Log only what you really need
- Logging should conform to values specified by online teams
- **Tools**: Thrift (used at Airbnb and Netflix)

## ⚡ Optimization Strategies for High-Volume Fact Data

### 1. Sampling

- Doesn't work for all use cases
- Works best for **metric-driven use cases** where imprecision isn't an issue

### 2. Bucketing

- Fact data bucketed by important dimension (usually user)
- **Bucket joins** = much faster than shuffle joins
- **Sorted-merge Bucket (SMB) joins** = joins without shuffle at all!

---

## 🗓️ Retention Strategies

### Big Tech Approach

| Table Size | Retention Strategy |
| --- | --- |
| **< 10 TB** | Retention doesn't matter much |
| **> 100 TB** | VERY SHORT RETENTION (~14 days or less) |

**Anonymization pattern**:

- Facts anonymized after 60-90 days
- Data moved to new table with PII stripped

**Why short retention for large tables**: High volumes make fact data costly to hold long-term

## 🔧 Deduplication of Fact Data

### The Problem

Facts can often be duplicated:

- You can click a notification multiple times
- Need to pick right deduplication window

### Questions to Ask

- No duplicates in a day? An hour? A week?
- **Best practice**: Look at distributions of duplicates

### Deduplication Options

**Intraday deduping**:

1. Streaming
2. Microbatch

---

## 🕐 Hourly Microbatch Dedupe Pattern

### Use Case

Reduce landing time of daily tables that dedupe slowly

### Real Example: Facebook Notifications

- Deduplicated **50 billion notification events every day**
- Reduced landing time from **9 hours after UTC** → **1 hour after UTC**

### How It Works

**Step 1**: Dedupe each hour with `GROUP BY`

**Step 2**: Dedupe between hours with `FULL OUTER JOIN` like branches of a tree

```
Hour 1  Hour 2  Hour 3  Hour 4
  |       |       |       |
  +-------+       +-------+
      |               |
      +---------------+
              |
          Final deduped
```

Pattern: hierarchical merging reduces total comparisons needed

## 💡 Key Takeaways

**Volume management**:

- Fact data is 10-100x dimension data volume
- Requires special optimization (sampling, bucketing, short retention)

**Denormalization trade-off**:

- Normalized = expensive joins on high-volume data
- Denormalized = faster queries, but data redundancy

**Deduplication is critical**:

- Facts duplicate more than dimensions
- Microbatch pattern enables fast daily deduping at scale

**Context through logging**:

- Collaborate with online engineers
- Log what you need, not everything
- Use schema standards (Thrift, Protobuf)
