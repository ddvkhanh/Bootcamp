### Core Concepts

- **Facts** are events that happened — login events, purchases, host price changes. They come from logs and are aggregated with `SUM`, `AVG`, `COUNT`. Higher volume than dimensions, but not always (rare events can be low-volume facts).
- **Dimensions** are snapshots of state — user's activation status, category, scoring tier. They appear in `GROUP BY` clauses. Can be high or low cardinality. Come from state, not events.
- **The test:** Did something *happen* (fact) or is it describing a *state* (dimension)?

> 💡 **Key distinction:** A login *event* is a fact. The derived daily state `dim_is_active` is a dimension built from those facts. You can aggregate facts into dimensions using `CASE WHEN` logic to bucketize values and reduce cardinality.
> 

---

### Deriving Dimensions from Facts

- Aggregate facts to create categorical dimensions — e.g. total purchases → `dim_scoring_class` (high/medium/low engager)
- Use `CASE WHEN` to bucketize continuous fact data into discrete tiers, reducing cardinality for analytics
- Example: Airbnb Superhost status is a dimension derived from booking counts, review scores, and response times (all facts)

### The Tricky Case — Airbnb Price

- **Question:** Is the price of a night on Airbnb a fact or dimension?
- It can be summed/averaged (sounds like a fact), hosts set the price (sounds like an event), and prices are doubles with extremely high cardinality
- **Answer:** The *fact* is the host changing the setting that impacted the price. The price itself is a *dimension* — it's derived from the state of the host's settings, not from a logged event.

> ⚠️ **Rule of thumb:** If a value is *derived from settings or config* rather than *logged as an event*, it's a dimension even if it looks aggregatable.
>

### Boolean & Existence Dimensions

- `dim_is_active`, `dim_bought_something` — daily or hourly grain dimensions derived from events
- `dim_has_ever_booked`, `dim_ever_labeled_fake` — "ever" dimensions flip once and never revert. Powerful ML features.
- Example: an Airbnb host with active listings who has *never* been booked looks sketchier over time
- `days_since_last_active`, `days_since_signup` — common in retention analysis and J-curve patterns

### The Date List Integer

- Extremely efficient data structure for tracking user growth and activity streaks
- Store a cumulated schema: `user_id`, `date`, `dates_active` (array of recent active days)
- Compress the array into an integer bitfield: `user_id`, `date`, `datelist_int`
- Example: `100000010000001` represents activity on specific days where `1` = active, `0` = inactive (bit position = day offset)
- Max Sung's writeup covers implementation details — see course resources

> ✅ **Efficiency gain:** Instead of storing arrays of dates per user, compress 30+ days of activity into a single integer column — massive storage and query performance improvement.
>

### State vs Activity Metrics

**dim_is_activated (state)**

- Snapshot of current status
- Measures total activated users
- Better for "signups" or cumulative growth

**dim_is_active (activity)**

- Derived from login events
- Measures daily/weekly active users
- Better for engagement and retention

*Great data science question: it depends on what you're measuring — growth (state) or engagement (activity).*
