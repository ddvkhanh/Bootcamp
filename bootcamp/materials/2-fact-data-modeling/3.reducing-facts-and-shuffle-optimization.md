### Why Minimize Shuffle?

- Big data leverages parallelism, but not all operations parallelize equally well
- **Highly parallel:** `SELECT`, `FROM`, `WHERE` — each worker processes independently
- **Kinda parallel:** `GROUP BY`, `JOIN`, `HAVING` — require coordination between workers
- **Painfully serial:** `ORDER BY` — must compare everything to produce global sort
    - this is different from a window function sort if we have partition by (then it will not be a global problem but will be a shuffle problem)

> 💡 **Key principle:** Shuffle happens when data must move across the network to different workers for grouping, joining, or sorting. Reducing shuffle means reducing network I/O and coordination overhead.
> 

---

### Making GROUP BY Efficient

- **Strategy 1:** Give `GROUP BY` buckets and guarantees — pre-partition data so each worker handles a known subset
- **Strategy 2:** Reduce data volume as much as possible *before* the shuffle happens
- Smaller input = less network transfer = faster query

### The Three Layers of Fact Data

| Layer | Schema | Volume | Grain |
| --- | --- | --- | --- |
| **Raw fact data** | `user_id, event_time, action, date` | Very high | 1 row per event |
| **Daily aggregate** | `user_id, action_cnt, date` | Medium | 1 row per user per day |
| **Reduced fact (long-array)** | `user_id, action_cnt_array, month_start` | Low | 1 row per user per month/year |

> ✅ **Volume reduction:** Compressing billions of raw events into millions of monthly array rows unlocks multi-year analyses that were previously impossible.
> 

---

### How Reduced Facts Work

Daily dates are stored as an **offset from a partition anchor** (e.g. `month_start` or `year_start`):

- **Index 0:** `month_start + 0 days` (July 1st)
- **Index 14:** `month_start + 14 days` (July 15th)
- **Last index:** `month_start + array_length - 1`

**Example:** For user 3 in July 2023, the array `[34, 3, 3, 4, 5, ...]` means:

- 34 likes on July 1st
- 3 likes on July 2nd
- 3 likes on July 3rd

---

### The SCD Accuracy Tradeoff

> ⚠️ **Critical tradeoff:** Dimensional joins become weird if you want performance. Your SCD accuracy collapses to the grain of the partition anchor — monthly or yearly snapshots, not daily precision.
> 
- You **give up 100% accurate SCD tracking** for massively increased performance
- Pick snapshots in time (month start, month end, or both) and treat dimensions as fixed within that period
- Example: If a user changed country mid-month, the reduced fact treats them as having one country for the entire month

**With daily SCD precision**

- Join raw facts to dimension daily
- Accurate to the day
- Multi-year query: hours to weeks

**With reduced facts (monthly snapshot)**

- Join array to dimension once per month
- Accurate to the month
- Multi-year query: minutes to hours

### Quick Reference
| Concept | When to Use | Watch Out For |
| --- | --- | --- |
| **Shuffle minimization** | High-volume fact queries with GROUP BY or JOIN | Pre-aggregate and bucket data before shuffle operations |
| **Daily aggregates** | Standard fact analytics, month-over-month reporting | Still high volume for multi-year queries |
| **Reduced facts** | Multi-year trend analysis, decade-long correlations | SCD accuracy collapses to monthly/yearly snapshots |
| **Array offset pattern** | Compressing time-series data into dense arrays | Index 0 = anchor date; must document offset calculation |
| **SCD snapshot tradeoff** | When performance matters more than daily precision | Dimensions fixed at month start/end — changes mid-period are ignored |
